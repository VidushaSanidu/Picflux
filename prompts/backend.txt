## Functional Requirements

### 3.1 Image Upload (Web)

- FR-1: The platform must allow a registered user to upload an image (Not forced) via the website.
- FR-2: On upload, the image must be stored with status = **Pending**.
- FR-3: Pending images must not be publicly visible.
- FR-4: The uploader must be able to see the status of their uploaded images.
- FR-5: Rate limit (5 per day (5 downloads & uploads per day))

**Upload metadata (minimum)**

- Title (optional)
- location (optional)
- Tags (optional)
- Uploaded by (user)
- Upload timestamp

### 3.2 Moderation and Verification (Admin)

- FR-5: Admin must be able to view a list of pending uploads.
- FR-6: Admin must be able to preview each pending image.
- FR-7: Admin must be able to **approve** an image.
    - Result: status becomes **Approved** and image becomes publicly visible.
- FR-8: Admin must be able to **reject** an image.
    - Result: status becomes **Rejected** and image remains non-public.
- FR-9: Admin should be able to provide an optional rejection reason visible to the uploader.

### 3.3 Public Browsing and Download (Web)

- FR-10: Public users must be able to browse approved images.
- FR-11: Public users must be able to search and filter approved images.
    - Minimum: search by title and tags.
- FR-12: Public users must be able to view an image detail page.
- FR-13: Public users must be able to download approved images. ( 5 downloads per day)
- need to get most common 5 tags

### 3.4 API Key Management

- FR-14: A registered user must be able to create an API key.
- FR-15: A registered user must be able to revoke/change an API key.
- FR-16: API keys must be stored securely (hashed at rest).

### 3.5 API: Upload Image

- FR-17: The API must allow authenticated upload of an animal image using an API key.
- FR-18: All API uploads must create images in **Pending** status.
- FR-19: API uploads must enforce a per-user daily limit of **5 uploads/day**.
- FR-20: If the daily upload limit is exceeded, the API must return a clear error response.

### 3.6 API: Download Image

- FR-21: The API must allow authenticated download of approved images using an API key.
- FR-22: API downloads must enforce a per-user daily limit of **5 downloads/day**.
- FR-23: If the daily download limit is exceeded, the API must return a clear error response.

### 3.7 Basic Abuse Prevention

- FR-24: The platform must validate that uploaded files are images.
- FR-25: The platform must limit maximum upload file size (10 MB). 
- FR-26: The platform should log upload and download events for auditing and rate limiting.

## 6. Key User Flows

### 6.1 Web Upload and Approval

1. User uploads an image.
2. Image status becomes Pending.
3. Admin reviews the pending queue.
4. Admin approves or rejects.
5. If approved, the image appears in public listings.

### 6.2 API Upload

1. User calls API with API key.
2. System verifies key and checks daily upload count.
3. If within limit, upload is accepted and stored as Pending.
4. Admin later approves to publish.

### 6.3 API Download

1. User requests an approved image download via API.
2. System verifies key and checks daily download count.
3. If within limit, download is allowed.

## 7. API Outline

> Exact paths and payloads can be adjusted during design.
> 

### Authentication

- Header: `Authorization: Bearer <api_key>`

### Endpoints

- `POST /api/v1/images` (upload)
- `GET /api/v1/images` (list approved images)
- `GET /api/v1/images/{id}` (image details)
- `GET /api/v1/images/{id}/download` (download)

### Rate limit responses

- Return `429 Too Many Requests` with a message indicating:
    - Which limit was exceeded
    - When the counter resets (start of next day)
